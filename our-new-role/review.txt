ARTICLE REVIEW
==============
"Utviklere! Hva blir vår rolle nå, når koding er løst?"
/ "Developers! What is our role now when coding is solved?"

Reviewed: 2026-03-15

---

OVERALL IMPRESSION
------------------
A strong, personal, and well-structured opinion piece that grounds the AI shift
in the author's own three-decade career arc. The historical framing — from
punched cards through assembler to AI agents — gives the argument depth and
avoids the hype-driven tone common in similar pieces. The writing is clear,
direct, and carries genuine conviction without being naive about the challenges.


STRENGTHS
---------
1. Personal credibility: The article draws on real experience (DevOps transition,
   PowerShell as "first digital colleague", hands-on Claude Code usage) which
   makes the claims feel earned rather than speculative.

2. Balanced perspective: It doesn't claim developers are obsolete — instead it
   reframes the role. The table mapping "Agent handles" vs "We must still own"
   is a particularly effective device.

3. Practical examples: The API log analysis scenario (check logs → find slow
   calls → create PR → notify Slack) makes the abstract concrete.

4. The "mob-instructing" concept: An original and actionable idea for how to
   onboard junior developers in an AI-first workflow. This is the most
   forward-thinking section.

5. Historical arc: The punched cards → assembler → OOP → AI agents progression
   reinforces the thesis that "the demand for precision has never gone away."

6. The boat/technical debt metaphor in the closing note is a charming personal
   touch that humanizes the piece.


ISSUES TO ADDRESS
-----------------
1. [INTENTIONAL] Java listed twice in tech stack paragraph (line 48 NO / line 40 EN):
   "Java eller C#, JavaScript-rammeverk — Python, Java eller Kotlin"
   → The first Java references the classic .NET vs Java divide; the second
   references the newer Java vs Kotlin debate within the JVM community.
   Intentional — if the reader knows, they know. No change needed.

2. Relative time references will age poorly:
   - "For omtrent to år siden" / "About two years ago" (re: Copilot testing)
   - "De siste par årene" / "The last couple of years"
   - "siste del av 2025 og starten på 2026"
   Consider pinning at least the Copilot start to a specific date, e.g.
   "In early 2024, my team and I started testing..."

3. The claim "Koding er rett og slett løst!" / "Coding is simply solved!" is the
   boldest statement in the article. It works as a provocative hook, but could
   benefit from a brief qualifier or a sentence acknowledging what coding problems
   remain hard for agents (e.g. novel algorithmic challenges, performance-critical
   systems, deeply domain-specific logic). Without it, experienced readers may
   dismiss the argument as overreach.

4. Blockquote formatting (NO version, lines 37-38): The two consecutive ">"
   lines render as a single blockquote in most Markdown renderers, which is the
   intent. But some parsers treat them as two separate paragraphs within the
   quote. Adding a ">" with blank line between them would ensure consistent
   rendering:
     > It's tempting to instruct: "Rewrite this old crappy code!"
     >
     > But a better approach is: ...

5. The English version is missing the byline and image references that the
   Norwegian .md has (author photo, regnbue.jpg, teknisk-gjeld.jpg, våken-katt.jpg,
   claude-pixel.svg). The EN .md reads as plain text without images, while the
   NO .md has full image references. These should be consistent.

6. Sources section: Only two references. Consider adding:
   - A link to Claude Code (since it's central to the narrative)
   - The GitHub Copilot research or changelog (since adoption is discussed)
   - Any internal or team reflections if publishable


LANGUAGE & TONE (Norwegian version)
------------------------------------
- Natural, conversational tone — reads well as a blog post / thought piece
- Good balance between technical specificity and accessibility
- Minor: "buzz" (line 46) is an anglicism — intentional and fine in a tech
  context, but worth noting if targeting a broader Norwegian audience


STRUCTURE
---------
The article flows well through these phases:
  1. Historical context (punched cards → assembler → today)
  2. Personal journey (evolution over 30 years)
  3. The recent revolution (Copilot → agents)
  4. Quality/technical debt angle
  5. Claude Code as turning point
  6. New competencies needed
  7. Knowledge transfer challenge
  8. Role redistribution (the table)
  9. Closing reflection

This is a logical and compelling arc. No structural changes needed.


RECOMMENDATIONS (priority order)
---------------------------------
1. Fix the Java duplicate — quick win, clear error
2. Add images/byline to the English .md for consistency
3. Add one qualifying sentence after "Coding is solved" claim
4. Pin the Copilot timeline to a specific date
5. Fix the blockquote formatting for cross-parser compatibility
6. [KEPT AS-IS] Two sources are sufficient — both are directly relevant and credible
